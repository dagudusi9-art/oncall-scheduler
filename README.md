# オンコール自動割当システム (oncall_scheduler)

外科のオンコール表(日中1名・夜間1名)を、各メンバーが入力した不都合日と
目標オンコール回数にもとづいて自動生成するシステムです。

- 最適化エンジン: Google OR-Tools (CP-SAT)
- 入力: ローカルCSV / Googleフォーム(→スプレッドシート) / Googleスプレッドシート直接入力
- 出力: Excel(.xlsx) / Googleスプレッドシートへの書き込み

## 2. セットアップ

### 2-1. Python環境

Python 3.10以上を推奨します。

```bash
cd oncall_scheduler
python3 -m venv venv
source venv/bin/activate        # Windowsの場合: venv\Scripts\activate
pip install -r requirements.txt
```

### 2-2. 動作確認(ローカルCSVのみで試す)

Google連携なしで、まずは動作確認ができます。

```bash
python -m src.main \
  --config config/config.yaml \
  --unavailability sample_data/unavailability.csv \
  --output output/schedule_2026_08.xlsx
```

`output/schedule_2026_08.xlsx` に「勤務表」「集計表」の2シートが出力されます。
うまく動けば、ターミナルに集計サマリーも表示されます。

## 3. Webアプリの使い方(推奨: 本番運用はこちら)

CLI版(`src/main.py`)に加えて、ブラウザからそのまま使えるWebアプリを
Streamlitで用意しています。管理者画面とメンバー画面が分かれており、
メンバーはスマホから不都合日をワンタップで入力できます。

> **補足**: `app/pages/` 以下のファイル名(`member_input.py`, `admin.py`)は
> 半角英数字(ASCII)にしています。Windowsの標準機能でzipを展開すると
> 日本語ファイル名が文字化けし、Streamlitがページを見つけられなくなる
> ことがあるためです。画面上に表示される「メンバー入力」「管理者」
> といった名前は `app/streamlit_app.py` 内で `st.Page(..., title="...")`
> として日本語で指定しているため、見た目には影響ありません。

### 3-1. 起動方法

```bash
cd oncall_scheduler
source venv/bin/activate
streamlit run app/streamlit_app.py
```

ブラウザで `http://localhost:8501` が開きます(社内サーバーで常時起動して
おけば、各医師がそのURL、例えば `http://hospital-oncall.example.com` に
スマホからアクセスして入力できます)。左のサイドバー、またはトップページの
ボタンから「メンバー入力」「管理者」の各画面に移動します。

### 3-2. ログイン(専用URL方式)

- **管理者**: 管理者画面に初めてアクセスすると、管理者パスワードの設定を求められます。以後はそのパスワードでログインします。
- **メンバー**: PINコードでの本人確認は廃止しました。代わりに、管理者画面
  「4. メンバー専用URLの発行」で、メンバーごとにランダムなトークン付きの
  専用URL(例: `.../member_input?token=otani_1a2b3c4d5e6f`)を発行します。
  管理者はこのURLをLINE等で1回だけ個別に送ってください。以後、メンバーは
  そのURLをブックマークしておくだけで、名前の選択やPINコードの入力なしに
  自分の入力画面へ直接アクセスできます。年月が変わってもURLは変わらないため、
  毎月同じURLをそのまま使い続けられます。
- URLを再発行すると古いURLは無効になります(トークンの漏洩が疑われる場合や、
  担当者が変わった場合などに使ってください)。

> **セキュリティに関する注意**: この専用URL方式は、医局内など限定された人数
> での運用を想定した簡易的な本人確認です。URLを知っている人なら誰でも
> アクセスできるため、パスワードのように厳重に管理するものではありませんが、
> 推測されにくいランダムな文字列になっています。病院外への公開や、極めて
> 機密性の高い情報の管理を想定した本格的な認証システムではありません。
> 院内ネットワーク限定での公開と併用することを推奨します。

### 3-3. メンバー入力画面

- 発行された専用URLでアクセスするだけで、自分の入力画面が開く(名前選択・ログイン操作は不要)
- 入力締切が設定されている場合、締切までの残り日数が表示される
- カレンダーが表示され、各日のマスをタップすると
  **○(終日OK) → ×(終日不可) → ▲昼(日中不可) → ▲夜(夜間不可) → ○...**
  の順に状態が切り替わる
- タップした瞬間に自動保存され、最終更新日時も記録される(保存ボタンを押し忘れる心配がない)
- 「入力内容を確定する」を押すと、(管理者が自動同期をONにしていれば)Googleスプレッドシートへも同期される
- LINEでの連絡や、管理者による転記作業は不要になる

この「ワンタップで状態が切り替わる」方式により、日中/夜間を別々に
選択するチェックボックスよりもスマホでの入力が速く行えます。

### 3-4. 管理者画面

1. **対象年月**: 対象となる年月を設定
2. **メンバー管理・目標回数**: 名前の追加・削除、メールアドレス・
   外部バイト対象の一覧編集。目標回数は通常「自動計算」され、終日不可が連続7日以上ある場合は、その期間を勤務可能日数から差し引いて
   自動的に調整される(詳細は「3-7. 目標回数の自動計算」を参照)
3. **外部バイト日の設定**: 外部病院バイトが必要な日をカレンダーからON/OFFで選択
   (詳細は「3-6. 外部病院バイトの自動割当」を参照)
4. **メンバー専用URLの発行**: メンバーごとの本人確認用URLを発行・再発行
   (「URLをコピー」の代わりに、コードブロック右上のコピーアイコンでコピーできます)
5. **入力締切・リマインダー**: 締切日の設定、入力状況一覧
   (入力済み/未入力/入力日数/最終更新日時)、未入力者へのリマインダー
   (LINE共有リンク・メール送信)
6. **Googleスプレッドシートへの自動同期**: メンバーが入力を確定するたびに
   自動でスプレッドシートへ反映する設定
7. **勤務表の作成・確定**:
   - 「勤務表を作成する(下書き)」で最適化を実行し、カレンダー表示・集計表・
     Excelダウンロードで内容を確認できる
   - 内容に問題がなければ「✅ この勤務表を確定する」をワンタップで実行すると、
     Excel・PDFが自動生成され、確定状態としてロックされる
     (以後は管理者が明示的に「確定を解除」しない限り上書きされない)
   - 確定後は「📱 LINEで勤務表確定をお知らせする」から共有文言付きでLINE共有できる

### 3-5. リマインダー・LINE共有について

- **リマインダー文章**: 未入力メンバーの一覧から自動生成され、コピーして
  使えます。「📱 LINEで共有する」ボタンでLINEアプリが開き、共有先を選んで
  そのまま送信できます(追加の契約・設定は不要です)。
- **メールでのリマインダー**: メンバーにメールアドレスを登録しておけば、
  SMTP設定(Gmail等)を入力してその場でメール送信もできます。
- **LINE Notifyについて**: 以前は「LINE Notify」というサービスで自動プッシュ
  通知ができましたが、2025年3月31日にサービス終了しています。自動での
  プッシュ通知をLINEで行いたい場合は、LINE公式アカウント + Messaging APIの
  契約が別途必要です(本システムの範囲外です)。本システムが提供するのは、
  追加契約が不要な「共有リンク」方式のみです。

### 3-7. 目標回数の自動計算

各メンバーの自院オンコール目標回数は、通常「自動計算」を利用します。

- **仕組み**: 「勤務可能日数(= その月の日数 - 長期不在日数)」の比率に応じて、
  その月のオンコール総数(日数×2)を各メンバーへ自動配分します。
  例: 31日間すべて勤務可能なら約8回、20日しか勤務できなければ約5〜6回、
  というように調整されます。
- **長期不在の設定**: 管理者画面「2. メンバー管理」の表で、メンバーごとに
  「長期不在開始」「長期不在終了」の日付を設定します。見学・長期休暇・病欠など、
  数日以上勤務できない期間を指定してください。
  (通常の「不都合日」は本人都合のため、この計算には影響しません)
- **手動指定**: 例外対応が必要な場合のみ、「手動指定」をONにすると
  「目標回数」を直接編集できるようになります。手動指定されたメンバーの回数は
  自動計算から除外され、残りの枠が他の自動計算メンバーに比率配分されます。
- 表には「勤務可能日数」「自動計算目標回数」が参考として表示されます
  (この2列は自動計算のため直接編集はできません)。

### 3-8. 外部病院バイトの自動割当

自院オンコール(日中・夜間)に加えて、外部病院バイトも同じ仕組みで
自動割当できます。希望制ではなく、対象者の中から自動的に割り当てられます。

- **対象者の設定**: 管理者画面「2. メンバー管理」の表で、メンバーごとに
  「外部バイト対象」のON/OFFを設定します(例: 8人中7人を対象にする)。
- **対象日の設定**: 管理者画面「3. 外部バイト日の設定」のカレンダーから、
  その月に外部バイトが必要な日をクリックしてON/OFFします。
- **割当条件**:
  - 対象日には、対象者の中から必ず1人が割り当てられます
  - メンバーが入力した不都合日(日中・夜間のどちらか一方でも不都合)がある日は、
    外部バイトも終日の勤務とみなして割り当てから除外します
  - 外部バイトに割り当てられた日は、その人は自院の日中・夜間どちらにも
    入りません
  - 外部バイトの回数は対象者の間でなるべく均等になるよう最適化します
    (自院オンコールの目標回数バランスとは別に調整されます)
- **表示・出力**: 勤務表には「日中」「夜間」「外部バイト」の3枠が表示され、
  集計表には「日中」「夜間」「自院合計」「外部バイト」「総勤務」が表示されます。
  Excel・PDF出力にも同様に反映されます。

### 3-9. Webアプリのデータ保存先

WebアプリはCLI版の `config.yaml` / CSVとは別に、`data/` フォルダに
JSON形式で設定・入力データを保存します(`data/config.json`,
`data/unavailability_2026_08.json`, `data/gaikobu_days_2026_08.json`,
`data/auth.json`, `data/finalized_2026_08.json`,
`data/last_updated_2026_08.json` など)。
このフォルダと `output/`(確定済みExcel・PDF)をバックアップしておけば、
そのままデータを保持できます。`data/auth.json` には管理者パスワードの
ハッシュ値と、各メンバーの専用URLトークンが入っているため、特に厳重に
管理してください(このファイルが漏洩すると全員のURLが漏れます。
その場合は管理者画面から該当メンバーのURLを再発行してください)。

### 3-10. 病院サーバーでの常時公開について

学内・院内サーバーやクラウド(例: AWS/GCPの小さいVMなど)で
`streamlit run app/streamlit_app.py --server.port 80` のように起動して
常時稼働させれば、各医師は特定のURL(例: `https://hospital-oncall.example.com`)
にスマホのブラウザでアクセスするだけで入力でき、LINEでのやり取りが不要になります。

社外に公開する場合は、院内ネットワーク限定にするなどセキュリティ設定を
別途検討してください。本システムの専用URL方式は限定的な人数での
運用を想定した簡易的なものなので、より高いセキュリティが必要な場合は、
HTTPS化(リバースプロキシ + 証明書)や、Basic認証・院内SSOとの併用を
推奨します。この「実際に外部からアクセスできるURLを用意する」作業
(サーバーの契約、ドメイン、HTTPS証明書の取得など)は、ネットワーク環境に
依存するため本システムのコードの範囲外です。管理者画面「4. メンバー専用
URLの発行」で設定する「アプリの公開URL」は、実際に取得したドメインに
合わせて更新してください。

### 3-11. 予定と実績の分離・年間実績の均等化・勤務交代

勤務表には「予定(scheduled_assignments)」と「実績(actual_assignments)」
の2種類があります。

- **予定**: 管理者画面「7. 勤務表の作成・確定」で自動生成し、確定した内容。
- **実績**: 確定した瞬間は予定と同じ内容でコピーされるが、その後は
  勤務交代の承認や管理者の手動修正によって、予定とは独立して更新される。
  **年間集計・翌月以降の均等化には必ず実績のみが使われます**(予定は
  一度だけ確定される「当初の割当」であり、その後の変化を反映しないため)。

**年間実績の均等化(自動割当への反映)**

管理者画面「7. 勤務表の作成・確定」で新しい月の勤務表を作成すると、
その年の「実績確定済み」の月の実績(自院オンコールの日中+夜間)を自動的に
参照し、年間実績が少ないメンバーをその月の割当で優先します。
自院オンコールは義務勤務であるため最優先の均等化対象とし、外部病院バイトは
義務ではないため対象者内での大きな偏りを避ける程度の副目的として扱っています
(詳細は「7. 割当ロジック」を参照)。

**勤務交代**

メンバー画面から、個人間で勤務(日中・夜間・外部バイト)を交代できます。

1. 交代したい本人が、メンバー画面「🔄 勤務交代」から交代依頼を作成する
   (交代してほしい勤務と、交代相手を選択)
2. 交代相手がメンバー画面から承認または却下する
3. 承認された時点で実績(actual_assignments)が即座に更新される
   (同日に他の枠を既に担当している場合など、矛盾が生じる場合は承認できません)
4. 管理者の承認は不要。管理者画面「10. 交代履歴・未処理の交代申請」で
   履歴の確認のみ行う

**実績の手動修正**

管理者画面「8. 実績勤務表・実績の手動修正」から、急な交代・病欠・
LINE上での交代済み・外部バイトキャンセルなど、勤務交代機能を使わない
ケースについて実績を直接修正できます。修正理由とともに履歴が保存されます。

**月末の実績確定**

管理者画面「11. 月末の実績確定」の「この月の実績を確定する」を押すと、
その月の実績が年間実績集計に反映されるようになります。確定後も管理者は
引き続き実績を修正できます(実績修正自体がもともと管理者専用の機能のため)。

## 4. 設定ファイル (config/config.yaml)


CLI版で使用する設定ファイルです(Webアプリを使う場合は管理者画面から
GUIで設定できるため、このファイルを直接編集する必要はありません)。

```yaml
year: 2026
month: 8

members:
  - name: 大谷
    target_count: 4
  - name: 中島
    target_count: 8
  ...
```

**重要**: `target_count` の合計は、その月の「日中+夜間の枠数の合計」
(= 日数 × 2)に近づけてください。合計が大きく異なると、目標回数から
外れる人が増えます(例: 31日の月なら枠数は62。8人で割るなら平均7〜8回)。

人数やメンバーは毎月自由に変更できます。ファイルを月ごとに複製しても、
`--config` オプションで別ファイルを指定してもかまいません。

## 5. 不都合日の入力方法

### 5-1. ローカルCSV(テスト・少人数向け)

`member_name,date,day_unavailable,night_unavailable` の列を持つCSVです。
`sample_data/unavailability.csv` を参考にしてください。

```csv
member_name,date,day_unavailable,night_unavailable
大谷,2026-08-01,1,0
大谷,2026-08-03,0,1
```

### 5-2. Googleフォーム + スプレッドシート(推奨: 本番運用)

各メンバーが自分のスマホ・PCから入力できるようにする方法です。

1. Googleフォームを作成し、次の3つの質問を設定する
   - **氏名**(プルダウン。メンバー一覧から選択させる)
   - **日中に不都合な日を選択してください**(チェックボックス。月内の全日付を選択肢に追加)
   - **夜間に不都合な日を選択してください**(チェックボックス。同上)
2. フォームの「回答」タブから、回答用スプレッドシートを作成する
3. そのスプレッドシートのIDを控える(URLの `.../d/【この部分】/edit`)
4. `src/forms_io.py` の `convert_form_responses()` で回答をロング形式に変換し、
   `src/sheets_io.py` の `_write_schedule_sheet` と同様の方法で
   「不都合日入力」シートへ書き出す運用にすると、`SheetsClient.load_unavailability()`
   がそのまま使えます。

   (フォームの質問文が異なる場合は `convert_form_responses()` の
   `name_column` / `day_unavailable_column` / `night_unavailable_column`
   引数を実際の列名に合わせて指定してください)

### 5-3. Googleスプレッドシートに直接入力する場合

フォームを使わず、共有スプレッドシートに直接ロング形式で入力してもらう
運用でも構いません。シート名「不都合日入力」に以下の列を用意してください。

| member_name | date | day_unavailable | night_unavailable |
|---|---|---|---|
| 大谷 | 2026-08-01 | 1 | 0 |

## 7. 割当ロジック(最適化の仕組み)

`src/optimizer.py` で OR-Tools CP-SAT を使ってモデル化しています。

### 絶対条件(必ず満たす / Hard constraints)

1. 不都合日には割り当てない(メンバーが入力した不都合日に加えて、
   長期不在期間も終日の不都合日として自動的に扱われる)
2. 各日・各枠(日中/夜間)には1人だけ割り当てる
3. 同じ人が同日の日中・夜間を両方担当しない
4. (外部バイト対象日) 外部バイト対象者の中から必ず1人を割り当てる
5. (外部バイト対象日) 不都合日(日中または夜間のどちらか)がある対象者には割り当てない
6. 外部バイトに割り当てられた日は、自院の日中・夜間どちらにも入らない

### できるだけ守りたい条件(ペナルティを最小化 / Soft constraints)

目的関数として以下の重み付き合計を最小化しています
(`OptimizerWeights` で調整可能。値が大きいほど重視。優先順位順に記載):

| 優先度 | 条件 | デフォルト重み | 説明 |
|---|---|---|---|
| 1 | 自院オンコールの年間実績均等化 | 300 | 「年間の確定済み実績」+「この月の割当」の合計の、メンバー間の最大-最小差(**最優先**) |
| 2 | 目標回数からのズレ | 50 | 各メンバーの自院オンコール合計と、その月の目標回数(自動計算 or 手動指定)の差の絶対値 |
| 3 | 連続勤務 | 30 | 夜間→翌日日中のような連続をペナルティ(過負荷防止のため強め) |
| 4 | 週末のペア化 | 20 | 土曜・日曜のオンコールが同じ週末にまとまらず分断された場合にペナルティ(できれば/公平性・連続勤務回避より優先度は低い) |
| 5 | 日中/夜間の偏り | 15 | 個人ごとの日中回数と夜間回数の差の絶対値 |
| 6 | 土日祝の偏り | 8 | 個人ごとの土日祝担当回数の、平均からのズレ |
| 7 | 外部バイトの偏り | 6 | 対象者間での外部バイト回数の最大-最小差(**義務ではないため最も優先度が低い**) |
| 8 | 勤務日の集中 | 5 | 近い日(3日以内)に複数回入ることをペナルティ |

自院オンコールは義務勤務であるため年間実績の均等化を最優先にしていますが、
外部病院バイトは義務ではないため、対象者内で大きな偏りが出ない程度の
副目的として扱っています。

**週末のペア化について**: 土日オンコールが月に2回ある場合、別々の週末に
1回ずつ入ると4週のうち2週末が潰れてしまいますが、同じ週の土曜・日曜に
まとめれば3週末を完全にフリーにできます。このためソフト制約として、
同じ人が「土曜だけ」「日曜だけ」に割り当てられる(=分断)ことをペナルティ化し、
なるべく同じ週末にまとまるようにしています。ただし絶対条件ではないため、
不可日・公平性(年間実績均等化・目標回数)・連続勤務回避の方が優先されます
(例: 週末をまとめるために夜間→翌日日中のような連続勤務が増えてしまう場合は、
連続勤務回避の重み(30)の方が週末ペア化(20)より大きいため、連続勤務回避が優先されます)。

重みを変更したい場合は `OptimizerOptions(weights=OptimizerWeights(...))` を
`main.py` や自作スクリプトから渡してください。外部バイトの対象日は
`OptimizerOptions(gaikobu_days={date(2026, 8, 1), ...})`、年間実績均等化の
入力は `OptimizerOptions(annual_actual_totals={"大谷": 12, ...})` のように渡します
(Webアプリ管理者画面から設定・実行した場合は自動的にこの形式に変換されます)。

外部バイトの目標回数([自院の目標回数]のような)個別設定は現状ありません。
対象者間でなるべく均等になるよう自動的に調整されます。

### 求解できない場合(INFEASIBLE)

絶対条件だけでどうしても矛盾する場合(例: ある日に全員が不都合、
外部バイト対象日にその日使える対象者が誰もいない、など)は
`status=INFEASIBLE` となり、Excelの「警告」シートに理由が出力されます。
その場合は不都合日の設定や目標回数、外部バイト対象日・対象者の設定を見直してください。

## 10. よくある質問

**Q. 目標回数の合計が枠数と一致しないとどうなりますか?**
A. Webアプリでは目標回数が自動計算されるため、通常は自動的に枠数
  (日数×2)と一致します。「手動指定」で複数人の回数を合計が枠数を
  超えるほど大きく設定した場合は、残りの自動計算対象メンバーの回数が
  0に近づく可能性があります。CLI版の `config.yaml` を直接編集する場合は、
  合計が枠数に近づくよう手動で調整してください。

**Q. メンバーの専用URLを忘れてしまった/漏洩が心配な場合は?**
A. 管理者画面「4. メンバー専用URLの発行」から、該当メンバーのURLを
  再発行してください。古いURLは自動的に無効になります。

**Q. 長期不在を設定すると、そのメンバーには何回もオンコールが回ってこなくなりますか?**
A. 目標回数(自動計算)が勤務可能日数に応じて少なくなるだけでなく、長期不在
  期間中は実際の割当からも自動的に除外されます(終日の不都合日として
  自動的に扱われるため、その期間に日中・夜間どちらも割り当てられません)。
  この期間について、あらためて「不都合日」を手動入力する必要はありません。

**Q. 祝日はどう設定しますか?**
A. `OptimizerOptions(holidays={date(2026, 8, 11), ...})` のように
  日付集合を渡してください。将来的には日本の祝日APIと連携する拡張も可能です。

**Q. Excelとスプレッドシート、どちらを使えばいいですか?**
A. 併用がおすすめです。Excelは配布・印刷用、スプレッドシートは
  常時共有・履歴確認用として使うと運用しやすくなります。
